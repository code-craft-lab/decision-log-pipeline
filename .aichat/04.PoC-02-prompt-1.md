=== KONTEKST BIZNESOWY ===
Chcemy zweryfikować tylko najbardziej krytyczną ścieżkę aplikacji OPA Decision Log Pipeline:
• Odbiór batcha decyzji z Open Policy Agent (OPA) przez HTTP.
• Publikacja odebranego batcha do Apache Kafka w formie JSON array.

=== ZAKRES PoC (MUST HAVE) ===
1. Endpoint HTTP:
   • POST /decision-logs
   • Przyjmuje dowolny JSON array (nie waliduj szczegółowo pól; traktuj payload jako String/JsonNode).
   • Zwraca HTTP 201 po *udanym* wysłaniu do Kafka, inaczej 500.

2. Producent Apache Kafka:
   • Łączy się z lokalnym brokerem Kafka (bootstrap servers z application.yml, domyślnie localhost:9092).
   • Publikuje cały otrzymany payload do tematu: decision-logs.
   • Klucz wiadomości – dowolny stały String, wartości – surowy payload JSON.

3. Obsługa błędów:
   • Jeśli Kafka zwróci wyjątek, endpoint zwraca 500 i loguje błąd.

4. Konfiguracja:
   • Profile spring-boot: local (domyślny) – tylko minimalne parametry do połączenia z Kafka.
   • Brak uwierzytelniania, chunkingu, DLQ, S3, OpenSearch, metryk itp.

5. Technologie (zgodnie z TECH-STACK):
   • Kotlin 2.x + Java 21
   • Spring Boot 3.5.8
   • Spring WebFlux (reactive)
   • Spring for Apache Kafka (kotlinx-reactive, jeśli konieczne)
   • Jackson
   • Gradle 8.8

6. Testy jednostkowe:
   • Minimalny test „happy path” dla kontrolera (WebTestClient) z użyciem embedded-Kafka lub mock Producer.

=== PO AKCEPTACJI ===
Po zatwierdzeniu planu przez użytkownika:
• Wygeneruj kompletny projekt Gradle (Kotlin DSL), plik settings.gradle.kts i README z instrukcją uruchomienia (docker-compose z jednym brokerem Kafka lub Otusa).
• Utwórz minimalne klasy, konfigurację, testy.
• Zapewnij, że projekt buduje się komendą `./gradlew clean build`.
